# 3.4 Persistent Data Management

Questa sezione descrive come il sistema UniMove gestisce i dati persistenti quindi quali dati vengono memorizzati, come vengono organizzati e le decisioni architetturali che motivano le scelte effettuate.

## Strategia di Persistenza

Il sistema utilizza un database relazionale **PostgreSQL 15** per la gestione dei dati persistenti. La scelta di un database relazionale è motivata da:

- **Integrità referenziale**: I vincoli di chiave esterna garantiscono la consistenza dei dati tra le tabelle correlate
- **Supporto nativo agli UUID**: PostgreSQL supporta nativamente il tipo `UUID`, utilizzato come chiave primaria in tutte le tabelle per motivi di sicurezza, evitando l'esposizione di identificatori sequenziali prevedibili nelle API
- **Maturità e affidabilità**: PostgreSQL è un DBMS consolidato, adatto a sistemi transazionali

## Gestione delle Migrazioni

La gestione dello schema del database è delegata interamente a **Flyway**. Hibernate è configurato in modalità `validate` infatti non crea né modifica tabelle automaticamente, ma verifica che lo schema esistente sia coerente con le entità JPA definite nel codice.

Ogni modifica al database deve essere tracciata tramite uno script di migrazione versionato nel formato:

```
VX__descrizione.sql
```

Gli script vengono eseguiti in ordine crescente di versione e tracciati nella tabella `flyway_schema_history`. Uno script già eseguito non viene mai modificato e in caso di correzione si crea un nuovo script con versione successiva.

Le migrazioni applicate al sistema sono le seguenti:

| Versione | Descrizione |
|---|---|
| `V1` | Schema iniziale: creazione delle tabelle `users`, `route_preferences`, `rides`, `bookings`, `reviews`, `notifications`. |
| `V2` | Aggiunta colonna `password_hash VARCHAR(255) NOT NULL DEFAULT 'CHANGEME'` alla tabella `users`. |
| `V3` | Modifica del tipo della colonna `travel_preferences` in `users` a `VARCHAR(500)`. |
| `V4` | Rimozione del vincolo `NOT NULL` e del valore di default dalla colonna `password_hash` in `users`, rendendola nullable. |
| `V5` | Aggiunta colonna `pers_id BIGINT` alla tabella `users`, necessaria per recuperare la foto profilo istituzionale da CINECA. |
| `V6` | Modifica del tipo della colonna `avatar_url` in `users` da `VARCHAR(500)` a `TEXT`, per supportare le immagini in formato base64. |
| `V7` | Aggiunta colonna `comment VARCHAR(500)` alla tabella `reviews`, rendendo il commento opzionale nelle recensioni. |


## Modello dei Dati

Il database è composto da sei tabelle principali:

| Tabella | Descrizione |
|---|---|
| `users` | Gli utenti del sistema quindi  studenti, docenti e dipendenti. Contiene username nel formato CINECA, email istituzionale costruita per convenzione, nome completo, ruolo, IBAN opzionale per ricevere donazioni, foto profilo in base64(`avatar_url`) e identificativo istituzionale CINECA per recuperare la foto profilo. Il campo `password_hash` è presente nella tabella ma è nullable, per gli utenti autenticati tramite CINECA non viene mai popolato. |
| `route_preferences` | Le tratte preferite impostate dall'utente. È una tabella separata con chiave esterna su `users` per evitare colonne ripetute e supportare fino a 3 tratte per utente. |
| `rides` | Le corse create dai guidatori. Contiene partenza, arrivo, fermate intermedie (3 hotspot massimo), dati del veicolo, numero di posti totali e disponibili e stato della corsa(default `OPEN`) e preferenze di viaggio del guidatore. |
| `bookings` | Le prenotazioni dei passeggeri. Collega un passeggero a una corsa tramite chiave esterna su `rides` e su `users`. Registra l'hotspot scelto dal passeggero e lo stato della prenotazione. Il vincolo `UNIQUE` garantisce che ogni passeggero possa prenotare una corsa al massimo una volta. |
| `reviews` | Le recensioni post-viaggio. Collegare recensore, recensito e corsa tramite chiavi esterne su `users` e `rides`. Contiene voto numero da 1 a 5 e un commento testuale opzionale. Il vincolo `UNIQUE` garantisce che ogni utente possa recensire il guidatore di una corsa al massimo una volta. |
| `notifications` | Le notifiche degli utenti: passaggio accettato, posto liberato, richiesta di passaggio e così via. |

## Decisioni Architetturali

- **UUID come chiave primaria**: Tutte le tabelle usano UUID invece di BIGSERIAL per evitare l'esposizione di identificatori sequenziali nelle API, aumentando la sicurezza del sistema.
- **`route_preferences` tabella separata**: Le tratte preferite sono in tabella separata con chiave esterna su `users` per evitare colonne ripetute e mantenere flessibilità nel numero di tratte per utente.
- **`ddl-auto=validate`**: Hibernate non gestisce lo schema, questa responsabilità è delegata interamente a Flyway, garantendo controllo e tracciabilità su ogni modifica al database.
- **Credenziali mai memorizzate**: Il sistema non salva mai le password degli utenti nel database. L'autenticazione è delegata a CINECA ESSE3.
- **Aggiornamento atomico dei posti disponibili**: Al momento della prenotazione e della cancellazione, il campo `available_seats` della tabella `rides` viene aggiornato nella stessa transazione della scrittura su `bookings`, garantendo consistenza dei dati in caso di accessi concorrenti.

## Query con filtri opzionali

La ricerca delle corse richiede di combinare più filtri opzionali quali username del guidatore, città di partenza, città di arrivo, data, in quasliasi combinazione scelta dall'utente. Spring Data Jpa non supporta nativamente la combinazione dinamica di filtri opzionali tramite query derivation(i metodi generati automaticamente dal nome).

Per questo motivo è stata adottata una query **JPQL personalizzata** tramite annotazione `@Query`, che applica ogni filtro solo se il parametro corrispondente non è `null`.

Questo approccio consente di gestire un numero arbitrario di filtri opzionali con una singola query, evitando la proliferazione di metodi repository per ogni combinazione possibile di filtri.

## Modalità di Accesso ai Dati

L'accesso ai dati persistenti avviene esclusivamente tramite il layer **repository** del backend, implementato con **Spring Data JPA**. Ogni tabella del database ha una corrispondente interfaccia repository che estende `JpaRepository`, delegando a Spring la generazione automatica delle query standard. Le entità JPA nel package `model` mappano direttamente le tabelle del database. Le API non espongono mai le entità direttamente infatti i dati transitano sempre attraverso oggetti DTO dedicati.

## Gestione del Lazy Loading

Le entità del modello utilizzano `FetchType.LAZY` per le relazioni `@ManyToOne`, per evitare di caricare dati non necessari ad ogni query. Questo richiede che l'accesso ai campi lazy avvenga all'interno di una sessione Hibernate attiva.
Per questo motivo, i metodi di servizio che convertono entità in DTO, dove vengono letti campi di relazione lazy, come `ride.getDriver().getUsername()` sono annotati come `@Transactional(readOlny = true)`. Senza ciò l'accesso al campo lazy fuori dalla sessione Hibernate genererebbe una `LazyInitializationException`.

> **Nota problema N+1:** L'attuale implementazione della ricerca corse presenta un problema di performance noto come **N+1 query**: per ogni `Ride` restituito dalla query principale, viene eseguita una query separata per caricare il `driver` associato. Con un numero elevato di risultati, questo comporta un numero proporzionale di query al database. Una soluzione futura prevede l'uso di `JOIN FETCH` nella query JPQL per caricare `Ride` e `User` in un'unica query.
