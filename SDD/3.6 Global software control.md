## 3.6 Global software control

Il flusso di controllo globale del sistema definisce come vengono sequenziate le operazioni e gestite le interazioni tra i sottosistemi. UniMove integra due diversi meccanismi di controllo in base al livello architetturale:

*   **Coordinamento Frontend/Backend:** Il client Flutter adotta un modello di controllo **Event-driven** basato sul pattern StateNotifier. Il sistema attende le interazioni dell'utente (es. inserimento credenziali o tap su un pulsante) che scatenano eventi gestiti asincronamente tramite la libreria `Dio` per invocare le API REST. Il backend Spring Boot adotta un modello **Thread-based**: per ogni richiesta HTTP in entrata viene assegnato un thread separato, garantendo la gestione parallela e concorrente delle richieste di più utenti.
*   **Gestione delle sessioni utente:** Il sistema opera in modalità completamente **stateless**. Non vengono mantenute sessioni lato server. Il controllo dell'identità avviene tramite un token **JWT**. Lato client, la gestione del token è affidata alla libreria `flutter_secure_storage`, che lo legge ad ogni richiesta per includerlo nell'header HTTP di autorizzazione.
*   **Gestione degli errori e delle eccezioni:** Per evitare risposte inconsistenti, la gestione delle eccezioni sul backend è centralizzata tramite un `GlobalExceptionHandler`. Questo componente intercetta gli errori generati dalla logica di business (es. corsa piena, utente non trovato) e li mappa in risposte HTTP standardizzate in formato JSON, garantendo che il client riceva messaggi uniformi e non vengano mai esposti dettagli tecnici o stack trace all'utente finale.
