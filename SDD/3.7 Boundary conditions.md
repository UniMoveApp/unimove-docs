## 3.7 Boundary conditions
Le condizioni limite descrivono il comportamento del sistema nelle fasi di non ordinaria amministrazione, come l'inizializzazione, lo spegnimento e la gestione dei fallimenti.
*   **Avvio del sistema (Startup):**
    - _Backend & Database:_ L'avvio dell'infrastruttura server prevede prima l'inizializzazione del container Docker di PostgreSQL, seguita dall'avvio dell'applicazione Spring Boot. Durante il bootstrap, il framework **Flyway** verifica ed esegue automaticamente gli script di migrazione SQL (`V1`, `V2`, `V3`) in combinazione con la politica di validazione di Hibernate (`hibernate.ddl-auto=validate`), garantendo che lo schema del database sia strutturalmente aggiornato, consistente e validato prima dell'accettazione del traffico.
	- _Client:_ All'avvio dell'applicazione mobile, il modulo di routing verifica localmente la presenza e la scadenza temporale (campo `exp` decodificato dal payload del JWT) del token precedentemente salvato. In caso di token valido, l'utente viene reindirizzato alla Home; in caso di token assente o scaduto, l'applicazione lo indirizza alla schermata di Login.
*   **Spegnimento del sistema (Shutdown):** Lato utente, l'arresto della sessione coincide con il *Logout*. Poiché il sistema è stateless, il logout consiste unicamente nell'eliminazione del token JWT dal *secure storage* del dispositivo mobile, senza necessità di notificare il server. 
*   **Gestione dei casi di errore (Fallimenti):**
    *   **Rete assente / Connessione persa:** Il client mobile deve gestire correttamente la perdita temporanea di connessione, evitando il blocco dell'applicazione o la perdita dei dati inseriti dall'utente (es. durante la creazione di una corsa).
    *   **Credenziali errate:** In caso di login fallito, il backend intercetta l'errore di validazione e restituisce uno status HTTP 401 (Unauthorized) con un messaggio di errore chiaro per l'utente, gestito opportunamente dal frontend.
    *   **Server esterno (CINECA) non raggiungibile:** Se il sistema di autenticazione istituzionale è in down temporaneo, il backend intercetta il timeout e restituisce all'app mobile un messaggio di errore esplicito sull'indisponibilità del servizio, proteggendo il sistema da crash incontrollati e senza esporre i dettagli tecnici della connessione fallita.
