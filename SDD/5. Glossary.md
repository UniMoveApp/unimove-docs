# 5. Glossary

Questa sezione fornisce le definizioni di tutti i termini specifici del dominio, gli acronimi e le abbreviazioni utilizzati all'interno del documento, al fine di garantire una comprensione univoca e condivisa del sistema tra tutti gli stakeholder del progetto.

- **Three-tier (Architettura a tre livelli):** Architettura software suddivisa in tre livelli logici (Presentation, Application, Data) che nel progetto separa nettamente il client mobile (Flutter), la logica applicativa (Spring Boot) e la gestione dei dati (PostgreSQL).
- **Controller (Control):** Oggetto di soluzione che riceve le richieste HTTP dal client Flutter, gestisce il flusso applicativo e le instrada al service corretto, senza mai contenere la logica di business.
- **Service:** Oggetto di soluzione, invocato dal Controller, che contiene, coordina e incapsula tutta la logica di business complessa dell'applicazione.
- **Repository:** Livello architetturale (implementato tramite interfacce Spring Data JPA) che incapsula la logica standard per interrogare e accedere ai dati persistenti del sistema.
- **Entity / Model:** Oggetti di applicazione (definiti come entità JPA) che rappresentano i concetti del dominio reale (es. Studente, Evento) e mappano direttamente le tabelle del database. Le entità non vengono mai esposte direttamente nelle API.
- **DTO (Data Transfer Object):** Oggetti utilizzati esclusivamente per trasportare i dati in entrata e in uscita dalle API REST o tra sottosistemi, al fine di nascondere l'implementazione interna e le entità del database.
- **COTS (Commercial Off-The-Shelf):** Componenti software commerciali o open-source già esistenti, collaudati e pronti all'uso (come PostgreSQL, Docker, Flyway) integrati nel sistema per favorirne il riuso.
- **Design Pattern:** Schema risolutivo generale, riutilizzabile e adattabile impiegato per risolvere problemi di progettazione ricorrenti.
- **Docker / Docker Compose:** Tecnologie di containerizzazione utilizzate per eseguire l'istanza del database PostgreSQL in locale. Docker Compose garantisce la riproducibilità e la consistenza dell'ambiente di sviluppo e testing tra i vari membri del team.
- **Flyway:** Strumento per la gestione versionata delle migrazioni del database. Ha la responsabilità esclusiva di tracciare e gestire ogni singola modifica allo schema del database.
- **Hibernate:** Framework ORM utilizzato dal backend. Nel progetto è configurato con il parametro `ddl-auto=validate` per delegare interamente la gestione e la modifica dello schema a Flyway.
- **JPA (Java Persistence API) / Spring Data JPA:** Interfacce utilizzate nel layer *repository* per la comunicazione con il database, delegando al framework Spring la generazione automatica delle query.
- **jjwt (v. 0.12.6):** Specifica libreria impiegata dal server backend per la generazione e la validazione dei token JWT necessari per la sicurezza.
- **Lombok:** Libreria utilizzata a *compile time* nel backend Spring Boot per automatizzare la generazione di codice boilerplate.
- **Maven:** Strumento di build automation e gestione delle dipendenze utilizzato per avviare il backend in fase di sviluppo.
- **RestClient:** Componente introdotto in Spring Framework 6 utilizzato per effettuare chiamate HTTP verso il sistema di autenticazione esterno.
- **GlobalExceptionHandler:** Meccanismo per la gestione centralizzata delle eccezioni, che garantisce risposte di errore controllate e uniformi su tutti gli endpoint REST.
- **Javadoc:** Strumento per generare e mantenere la documentazione HTML direttamente tramite i commenti nel codice sorgente, illustrando responsabilità, parametri, eccezioni e valori di ritorno delle classi.
- **Basic Auth:** Metodo di autenticazione HTTP utilizzato dal backend per comunicare e validare le credenziali sull'endpoint del sistema esterno.
- **CINECA / ESSE3:** CINECA è il Consorzio Interuniversitario che gestisce ESSE3, il sistema informativo e gestionale accademico di UNIMOL. UniMove lo utilizza come *Identity Provider* (*source of truth*) per autenticare il ruolo e l'identità degli utenti, senza mai salvare le password nel proprio database.
- **UUID (Universally Unique Identifier):** Tipo di dato supportato nativamente da PostgreSQL, utilizzato come chiave primaria in tutte le tabelle del database. È preferito ai tipi sequenziali come `BIGSERIAL` per questioni di sicurezza, evitando di esporre ID prevedibili nelle API pubbliche.
- **IBAN (International Bank Account Number):** Codice bancario standard che gli utenti guidatori possono opzionalmente inserire per ricevere le donazioni per le spese di viaggio.
- **GDPR e PSD2:** Rispettivamente il Regolamento UE sulla privacy per il corretto trattamento dei dati sensibili e di geolocalizzazione e la Direttiva UE sui servizi di pagamento, rilevante per le donazioni inserite nell'applicazione.
- **ODD (Object Design Document):** Documento tecnico che illustra in dettaglio le decisioni di design, rendendo esplicite le interfacce, i contratti, le classi, le visibilità e i pattern adottati tra i sottosistemi.
- **RAD (Requirements Analysis Document):** Documento di analisi che descrive in dettaglio il sistema in termini di modelli, scenari e requisiti funzionali e non funzionali. Funge da contratto base tra clienti e sviluppatori.
- **Information Hiding (Principio Need to Know):** Principio di design adottato secondo cui le interfacce pubbliche devono esporre unicamente ciò che serve al chiamante, proteggendo gli attributi interni e i dettagli implementativi.
