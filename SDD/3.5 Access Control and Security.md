## 3.5 Access Control and Security

Questa sezione descrive le politiche di controllo degli accessi e le misure di sicurezza adottate nel sistema UniMove.

## Autenticazione

L'autenticazione degli utenti è delegata interamente al sistema istituzionale **CINECA**. Al momento del login, le credenziali inserite dall'utente vengono validate tramite una chiamata all'endpoint `/e3rest/api/login` delle API ESSE3 con Basic Auth.

In caso di autenticazione riuscita:

- L'utente viene creato nel database locale se non esiste ancora.
- I dati dell'utente vengono aggiornati ad ogni login, infatti CINECA è la nostra **source of truth**.
- L'email istituzionale viene costruita per convenzione: `username@studenti.unimol.it` per studenti e `username@unimol.it` per docenti e dipendenti.
- Il ruolo viene mappato dal campo `grpDes` restituito da CINECA verso l'enum interno: `STUDENT`, `PROFESSOR`, `STAFF`

## Token JWT

Al termine del login, il backend genera un token **JWT** firmato con algoritmo **HMAC-SHA384 (HS384)** tramite libreria jjwt 0.12.6. Il token viene restituito al client mobile e utilizzato per autenticare tutte le richieste successive.

Caratteristiche del token:

- **Durata**: 10 giorni(864000000 ms)
- **Contenuto**: username e ruolo utente
- **Trasmissione**: header HTTP `Authorization: Bearer <token>`
- **Stateless**: Il backend non mantiene sessioni server-side, ogni richiesta viene validata indipendentemente tramite token

La scelta di HMAC-SHA384 è motivata dall'architettura monolitica del backend, in un sistema distribuito sarebbe preferibile RSA per consentire verifica del token senza condividere chiave segreta. L'algoritmo HS384 viene scelto automaticamente da jjwt 0.12.x come algoritmo minimo sicuro per chiavi di 256 bit.

## Controllo degli Accessi

Spring Security è attivo dal primo avvio del sistema. La configurazione adottata è la seguente:

- **CSRF disabilitato**: Il sistema utilizza JWT stateless invece di sessioni server-side, rendendo la protezione CSRF non necessaria.
- **Solo `/api/auth/**` è pubblico**: Tutti gli endpoint sotto il path `/api/auth/` sono accessibili senza autenticazione, gli altri richiedono un token jWT valido.
- **Nessun ID esposto nella request**: L'identità dell'utente viene ricavata esclusivamente dal token JWT, non da parametri nella request.

## JwtFilter

Il `JwtFilter` è un filtro di Spring che intercetta ogni richiesta HTTP in entrata. Per ogni richiesta:

1. Estra il token dall'header `Authorization`
2. Valida il token tramite `JwtUtil`
3. Popola il `SecurityContext` di Spring con identità dell'utente.

> **Nota limitazione:** Il `JwtFilter` popola il `SecurityContext` con il 
> ruolo fisso `ROLE_USER` per tutti gli utenti, indipendentemente dal ruolo reale 
> memorizzato nel token JWT. Non esistono ancora endpoint con autorizzazioni differenziate 
> per ruolo. La gestione granulare dei ruoli verrà implementata in uno sprint successivo.

## Gestione delle Eccezioni

Il `GlobalExceptionHandler` gestisce centralmente tutte le eccezioni sollevate dalla logica di business, garantendo risposte HTTP semanticamente corrette e uniformi su tutti gli endpoint REST. Sono definite le seguenti categorie di errore:

| Eccezione | HTTP Status | Descrizione |
|---|---|---|
| `MethodArgumentNotValidException` | `400 Bad Request` | Dati in ingresso non validi secondo i vincoli definiti nei DTO. Restituisce il dettaglio dei campi non validi. |
| `InvalidRequestException` | `400 Bad Request` | Richiesta semanticamente errata dalla logica di business (es. cancellazione oltre il limite di 24 ore, prenotazione su corsa piena). |
| `AuthFailedException` | `401 Unauthorized` | Autenticazione fallita (es. credenziali CINECA non valide). |
| `UnauthorizedException` | `403 Forbidden` | Operazione non consentita all'utente autenticato (es. tentativo di modificare una risorsa altrui). |
| `ResourceNotFoundException` | `404 Not Found` | Risorsa richiesta non esistente nel sistema (es. corsa o prenotazione non trovata). |
| `RuntimeException` | `401 Unauthorized` | Fallback generico per eccezioni non gestite esplicitamente. |

Le eccezioni custom estendono `RuntimeException` e vengono lanciate dai service quando
si verifica una condizione di errore applicativo. Il `GlobalExceptionHandler` le
intercetta tramite `@ExceptionHandler` e le mappa nella risposta HTTP corrispondente,
senza mai esporre stack trace o dettagli tecnici interni al client.


## Gestione del Token lato Client

Il client mobile Flutter gestisce il token JWT tramite la libreria **flutter_secure_storage**, che utilizza il sistema di archiviazione sicura del dispositivo (Keychain su iOS, Keystore su Android).

Le operazioni gestite da `AuthService` sono:

| Operazione | Descrizione |
|---|---|
| **Salvataggio** | Il token viene salvato in secure storage al termine del login con chiave `jwt_token` |
| **Lettura** | Il token viene letto ad ogni richiesta autenticata e incluso nell'header `Authorization: Bearer <token>` |
| **Verifica scadenza** | All'avvio dell'app, il token viene decodificato lato client per verificare il campo `exp`, se scaduto l'utente viene reindirizzato al login |
| **Eliminazione** | Al logout, il token viene eliminato dal secure storage tramite `deleteToken()`, nessuna chiamata fatta al backend |

La verifica della scadenza avviene tramite la libreria **dart_jsonwebtoken** che decodifica il token senza verificare la firma — serve solo per leggere il campo `exp` e confrontarlo con la data corrente.
