# 3.3 NonFunctional Requirements

---

## 3.3.1 Usability

Il sistema deve essere utilizzabile senza necessità di training preliminare da parte dell'utente. Essendo un'applicazione destinata a studenti, docenti e dipendenti universitari con diversi livelli di familiarità tecnologica, l'interfaccia deve risultare intuitiva e accessibile a tutti.

- L'applicazione deve consentire a un nuovo utente di completare il primo accesso e configurare le proprie preferenze entro 3 minuti dal primo avvio.
- I messaggi di errore devono essere chiari, in lingua italiana e devono indicare all'utente come procedere per risolvere il problema.
- La navigazione tra le schermate principali deve essere raggiungibile con un massimo di 3 tap dalla schermata home.
- L'interfaccia deve adattarsi alle dimensioni dello schermo del dispositivo mobile dell'utente(responsive layout)

---

## 3.3.2 Reliability

Il sistema deve garantire un livello di affidabilità adeguato all'utilizzo quotidiano da parte della comunità accademica UNIMOL.

- Il sistema deve garantire una disponibilità minima del 95% su base mensile.
- In caso di errore lato server, il sistema deve restituire un messaggio di errore appropriato senza esporre dettagli tecnici interni all'utente.
- Il sistema deve gestire correttamente la perdita temporanea di connessione del dispositivo mobile, senza perdita di dati inseriti dall'utente.
- Le operazioni critiche come la prenotazione di una corsa devono essere atomiche, in caso di errore il sistema deve ripristinare lo stato precedente.
- In caso di irraggiungibilità temporanea del servizio di autenticazione CINECA ESSE3, il sistema deve restituire all'utente un messaggio di errore chiaro che indichi l'indisponibilità del servizio, senza esporre dettagli tecnici interni.

---

## 3.3.3 Performance

Il sistema deve garantire tempi di risposta accettabili anche in condizioni di carico normale, considerando che gli utenti accedono prevalentemente nelle fasce orarie mattutine e preserali tipiche degli spostamenti universitari.

- Le API del backend devono rispondere entro 2 secondi per il 95% delle richieste in condizioni di carico normale.
- Il sistema deve supportare almeno 100 utenti concorrenti senza degradazione delle prestazioni.
- Il caricamento della schermata home dopo il login non deve superare i 3 secondi in condizioni di buona rete.
- Le operazioni di ricerca corsa devono restituire i risultati entro 2 secondi dalla sottomissione della query.

---

## 3.3.4 Supportability

Il sistema deve essere progettato per facilitare la manutenzione, l'aggiornamento e l'evoluzione nel tempo.

**Architettura**
Il sistema è organizzato secondo un'architettura a tre livelli che separa nettamente il client mobile (Flutter), la logica applicativa (Spring Boot) e la gestione dei dati (PostgreSQL). Questa separazione consente di modificare o aggiornare ciascun livello in modo indipendente senza impattare gli altri.

**Manutenibilità**
 Il codice sorgente del backend deve essere organizzato secondo una struttura a package ben definita e documentata, che separi chiaramente le varie responsabilità. Le modifiche al Database devono essere gestite tramite script di migrazione versionati(Flyway), garantendo tracciabilità e riproducibilità dell'ambiente in qualsiasi momento.

 **Portabilità**
 L'utilizzo di Docker Compose per l'orchestrazione del database garantisce che l'ambiente di sviluppo sia riproducibile su qualsiasi macchina del team, indipendentemente dal sistema operativo.

**Operabilità**
 Il sistema deve produrre log applicativi sufficienti a diagnosticare errori in produzione senza richiedere accesso diretto al codice sorgente. La gestione centralizzata delle eccezioni tramite GlobalExceptionHandler garantisce risposte di errore uniformi e controllate su tutti gli endpoint.

---

## 3.3.5 Implementation

Il sistema è soggetto ai seguenti vincoli implementativi:

- Il client deve essere un'applicazione mobile sviluppata con il framework Flutter, per garantire compatibilità cross-platform con dispositivi Android e IOS.
- Il backend deve esporre esclusivamente API REST, consumate dal client mobile tramite chiamate HTTP.
- Il linguaggio di programmazione del backend è Java 21 LTS.
- Il framework di backend è Spring Boot 4.0.6.
- Il database relazionale utilizzato è PostgreSQL 15.
- La gestione delle migrazioni del database è affidata a Flyway, ogni modifica allo schema deve essere tracciata tramite script di versionamento.
- Le chiavi primarie di tutte le tabelle del database devono essere di tipo UUID, per motivi di sicurezza e per evitare l'esposizione di identificatori sequenziali nelle API.

---

## 3.3.6 Interface 

Il sistema è soggetto ai seguenti vincoli di interfaccia:

- La comunicazione tra client Flutter e backend Spring Boot avviene esclusivamente tramite API REST in formato JSON.
- Ogni richiesta al backend, ad eccezione dell'endpoint di login, deve includere un token JWT valido nell'header HTTP Authorization con prefisso Bearer.
- L'autenticazione degli utenti avviene tramite credenziali istituzionali UNIMOL, validate tramite API ESSE3 di Cineca tramite Basic Auth sull'endpoint `/e3rest/api/login`.
- Il backend non espone mai direttamente le entità del database, tutti i dati transitano tramite DTO.

---

## 3.3.7 Packaging

- L'applicazione mobile è distribuita tramite installazione diretta del pacchetto APK su dispositivi Android per le fasi di sviluppo e testing.
- Il database PostgreSQL viene eseguito in locale tramite un container Docker gestito da Docker Compose. Il backend Spring Boot viene avviato separatamente tramite Maven o dall'IDE IntelliJ IDEA.
- L'ambiente di sviluppo deve essere riproducibile su qualsiasi macchina del team tramite i file di configurazione Docker Compose presenti nel repository.

---

## 3.3.8 Legal

Il sistema deve essere conforme alle seguenti normative:

- **GDPR (Regolamento UE 2016/679):** Il sistema deve garantire la conformità al GDPR per il trattamento dei dati personali degli utenti, in particolare per i dati di geolocalizzazione GPS acquisiti durante la corsa, i dati bancari (IBAN) utilizzati per le donazioni volontarie e i contenuti delle chat private, che devono essere eliminati al termine della corsa.
- **Direttiva PSD2 (Direttiva UE 2015/2366):** Il sistema deve rispettare la normativa europea sui servizi di pagamento digitale per la gestione delle donazioni volontarie effettuate tramite l'applicazione.
- **Normative sul Car Pooling:** Il sistema deve rispettare la normativa italiana sulla mobilità condivisa senza scopo di lucro. Il viaggio rimane gratuito — le quote di supporto rappresentano esclusivamente un contributo volontario alla copertura parziale delle spese del veicolo.
- **Privacy Policy e Condizioni d'Uso:** Il sistema deve richiedere l'accettazione esplicita della Privacy Policy e delle Condizioni d'Uso prima di consentire l'accesso all'applicazione.
