# 2. Packages

Questa sezione descrive l'organizzazione dei package del sistema UniMove, documentando le responsabilità di ciascun componente architetturale e le direzioni delle dipendenze che regolano il flusso dei dati.

## Overview 

L'organizzazione dei package di UniMove descrive come il codebase è strutturato a livello architetturale. Il sistema è composto da due codebase principali:

```text
UniMove

├── Flutter Mobile Frontend
└── Java Spring Boot Backend
```

Il Flutter Mobile Frontend rappresenta il client mobile cross-platform per Android e IOS, organizzato sotto la directory `lib`.

Il Java Spring Boot Backend rappresenta il server che espone le API REST, coordina la logica applicativa, gestisce l'autenticazione tramite CINECA ESSE3 e la persistenza dei dati tramite PostgreSQL.

L'organizzazione dei package segue un approccio modulare e stratificato, separando le responsabilità per migliorare la manutenibilità e ridurre l'accoppiamento tra i componenti.

## Flutter Mobile Frontend Packages

Il frontend mobile è organizzato sotto la directory `lib` e segue una struttura modulare che separa elementi core, widget condivisi e moduli specifici per feature.

Struttura ad alto livello:

```text
lib
├── core
├── features
├── shared
├── app.dart
└── main.dart
```

### Core Package

Il package `core` contiene elementi riutilizzabili non specifici di una singola feature.

```text
core
├── constants
├── router
└── services
```

Il package `constants` contiene le costanti globali dell'applicazione, tra cui le costanti per le API.

Il package `router` contiene la struttura di navigazione dell'applicazione tramite la libreria `go_router`. Il file `app_router.dart` definisce tutte le rotte dell'app (`/splash`, `/login`, `/benvenuto`, `/home`, `/esplora`, `/corse/crea`, `/corse/cerca`, `/profilo`) e gestisce il redirect automatico in base allo stato di autenticazione — se l'utente non è autenticato viene reindirizzato a `/login`, se è autenticato ma non ha completato l'onboarding viene reindirizzato a `/benvenuto`.

Il package `services` contiene i servizi condivisi tra le feature: `AuthService` per la gestione del token JWT tramite `flutter_secure_storage` e la verifica dello stato di onboarding, e `ApiClient` per le chiamate HTTP tramite la libreria `Dio`.

### Shared Package

Il package `shared` contiene elementi riutilizzabili condivisi tra più feature.

```text
shared
├── theme
└── widgets
```

Il package `theme` contiene la configurazione visiva dell'app — colori, tipografia e impostazioni generali del tema.

Il package `widgets` contiene i componenti grafici comuni: `MainScaffold` fornisce la struttura comune dell'app con AppBar e BottomNavigationBar a 5 voci (Home, Esplora, Crea Corsa, Cerca Corsa, Profilo), e `skeleton.dart` fornisce i placeholder di caricamento.

### Features Package

Il package `features` contiene le aree funzionali principali dell'applicazione mobile.

```text
features
├── auth
├── home
├── profile
└── rides
```

### Auth Feature Package

```text
auth
└── presentation
    ├── auth_controller.dart
    ├── auth_test_screen.dart
    ├── login_screen.dart
    └── welcome_routes_screen.dart
```

Il layer `presentation` contiene la schermata di login (`LoginScreen`), la schermata di benvenuto per la selezione delle tratte preferite (`WelcomeRoutesScreen` — permette di inserire fino a 3 tratte con possibilità di saltare, salva il completamento dell'onboarding tramite `AuthService` ma non persiste ancora le tratte nel backend), il controller di stato (`AuthController`) basato su Riverpod `StateNotifier`, e una schermata di test per l'autenticazione.

### Home Feature Package

```text
home
└── presentation
    └── home_screen.dart
```

Contiene la schermata principale dell'applicazione con header utente, card promemoria eventi e tre tab: "I miei eventi" (con azioni Avvia/Elimina), "Prenotazioni" (con azioni Traccia/Abbandona) e "Archivio" (con azioni Dona/Recensisci). I dati degli eventi sono attualmente mock — l'integrazione con le API backend verrà completata nei sprint successivi.


### Profile Feature Package

```text
profile
└── presentation
    └── profile_screen.dart
```

Contiene la schermata del profilo utente.

### Rides Feature Package

```text
rides
└── presentation
    ├── create_ride_screen.dart
    └── search_ride_screen.dart
```

Contiene le schermate per la creazione e la ricerca delle corse.

### Flutter Mobile Package Dependencies

La direzione delle dipendenze nel frontend mobile è:

```text
features
↓
shared
↓
core
```

I package delle feature possono usare widget riutilizzabili da `shared` e servizi comuni da `core`. Il package `shared` deve rimanere indipendente dalla logica specifica delle feature. Il package `core` fornisce infrastruttura comune e non dipende da package specifici delle feature.

## Java Spring Boot Backend Packages

Il backend è organizzato sotto il package `com.unimove.unimove` e segue un'architettura a layer che separa l'esposizione delle API REST, la logica applicativa, la persistenze dei dati e la sicurezza.

Struttura ad alto livello:

```text
com.unimove.unimove
├── config
├── controller
├── dto
│   ├── request
│   └── response
├── exception
├── model
├── repository
├── security
└── service
```

### Controller package

Il package `controller` contiene i punti di ingresso REST esposti dal backend.

```text
controller
```

Questo package è responsabile della ricezione delle richieste HTTP dai client Flutter e della restituzione delle risposte HTTP. I Controller non contengono la logica di business, il loro ruolo è esporre API e delegare le operazioni al Service.
Attualmente sono presenti `AuthController` per gli endpoint di autenticazione (`/api/auth/**`) e `UserController` per gli endpoint del profilo utente (`api/users/**`)

### Service package

Il package `service` contiene la logica di business del backend.

```text
service
```

Questo package coordina le operazioni principali del backend, tra cui autenticazione tramite CINECA ESSE3, gestione profilo utente e la validazione delle tratte preferite. I service possono utilizzare repository, DTO, elementi di configurazione e il sistema di sicurezza.
Attualmente sono presenti `AuthService` che gestisce l'autenticazione tramite CINECA ESSE3 e la generazione del token JWT e `UserService` che gestisce il profilo utente, le preferenze di viaggio, le informazioni bancarie e le tratte preferite.

### Repository package

Il package `repository` contiene le interfacce per l'accesso ai dati persistenti.

```text
repository
```

Ogni interfaccia estende `JpaRepository` di Spring Data JPA, delegando a Spring la generazione automatica delle query standard. Attualmente sono presenti `UserRepository` e `RoutePreferenceRepository`.

### Model package

Il package `model` contiene le entità JPA che mappano le tabelle del database.

```text
model
```

Attualmente sono presenti `User` e `RoutePreference`, oltre all'enum `Role` (STUDENT, PROFESSOR, STAFF). Tutte le entità utilizzano UUID come chiave primaria.

### DTO package

Il package `dto` è suddiviso in due sottopacchetti:

```text
dto
├── request
└── response
```

Il package `request` contiene: `LoginRequest`, `RoutePreferenceRequest`, `UpdateIbanRequest`, `UpdatePreferenceRequest`. Il package `response` contiene: `LoginResponse`, `UserProfileResponse`, `RoutePreferenceResponse`, `CinecaLoginResponse`. Le entità del model non vengono mai esposte direttamente nelle API.

### Config package

Il package `config` contiene la configurazione di Spring Security e CORS.

```text
config
```

Contiene `SecurityConfig` che configura Spring Security, disabilita CSRF, imposta le sessioni come stateless, definisce endpoint pubblici(`/api/auth/**`) e aggiunge il `JwtFilter` prima del filtro di autenticazione standard.

### Security package

Il package `security` contiene il filtro JWT e le classe specifiche per autenticazione.

```text
security
```

Contiene `JwtUtil` per la generazione e validazione dei token JWT tramite HMAC-SHA384 e `JwtFilter` che intercetta ogni richiesta HTTP, estrae il token dall'header `Authorization: Bearer` e popola il `SecurityContext` di Spring. `JwtFilter` dipende da `JwtUtil` tramite constructor injection.

### Exception package

Il package `exception` contiene la gestione centralizzata degli errori.

```text
exception
```

Contiene il `GlobalExceptionHandler` che intercetta le eccezioni generate dalla logica di business e la mappa in risposte HTTP standardizzate in formato JSON, garantendo che il client riceva messaggi di errore uniformi senza esporre dettagli tecnici interni.

### Backend Package Dependencies

La direzione delle dipendenze nel backend è:

```text
controller
↓
service
↓
repository
↓
database
```

I controller dipendono dai service per eseguire operazioni applicative. I service coordinano la logica di business e possono dipendere da repository, DTO, config e security. I repository gestiscono l'accesso ai dati persistenti. I DTO sono utilizzati tra controller e service per scambiare dati senza esporre entità interne.

```text
service
↓
CINECA ESSE3 (sistema esterno)
```

La comunicazione con CINECA ESSE3 avviene tramite `RestClient` di Spring Framework, isolata nel service layer per evitare dipendenze dirette dai controller


## Dipendenze Globali tra i Sistemi

Le dipendenze globali definiscono come il frontend Flutter e il backend Spring Boot comunicano tra loro.

```text
Flutter Mobile Frontend
        ↓
Java Spring Boot Backend REST APIs
        ↓
Backend Services
        ↓
Repository / Database
```

Il client Flutter comunica con il backend esclusivamente tramite API REST in formato JSON. Il client non accede mai direttamente al database e non comunica direttamente con sistemi esterni come CINECA ESSE3.

L'integrazione con CINECA ESSE3 è isolata nel backend:

```text
Backend Service
        ↓
CINECA ESSE3 (sistema esterno)
```

Questo mantiene i dettagli dell'integrazione esterna nascosti al client mobile e separati dal layer di dominio del backend.
