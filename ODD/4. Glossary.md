# 4. Glossario

Questa sezione fornisce le definizioni formali dei concetti e dei termini tecnici specifici dell'Object Design utilizzati all'interno del documento, al fine di garantire una comprensione condivisa delle scelte architetturali tra sviluppatori e tester.

*   **API (Application Programming Interface):** Insieme di procedure e strumenti che permettono la comunicazione tra il client mobile e il backend. Nel layer Controller, queste si traducono in endpoint HTTP REST.
*   **Controller (Control Layer):** Oggetto di soluzione che funge da strato di orchestrazione leggero. Riceve le richieste HTTP, estrae l'identità dell'utente (tramite il Principal) e delega l'esecuzione della logica applicativa al Service corrispondente, mappando infine le risposte.
*   **COTS (Commercial Off-The-Shelf):** Componenti software o librerie già esistenti (come Spring Boot, Flutter o la libreria jjwt) che vengono integrati nel sistema per favorirne il riuso (Buy vs. Build).
*   **Design by Contract:** Principio di progettazione adottato per tutte le interfacce pubbliche, che prevede la definizione esplicita di *precondizioni* (cosa deve essere vero prima), *postcondizioni* (cosa è garantito dopo) ed *eccezioni* per ogni operazione.
*   **Design Pattern:** Schema risolutivo generale, riutilizzabile e adattabile impiegato per risolvere un problema di progettazione ricorrente nel software.
*   **DTO (Data Transfer Object):** Oggetti di soluzione utilizzati esclusivamente per trasportare i dati in entrata e in uscita dal sistema, nascondendo la complessità interna dell'albero degli oggetti di dominio.
*   **Entity:** Oggetto di applicazione che rappresenta i concetti persistenti del dominio reale (es. User, Ride) e che mappa direttamente le informazioni nel database.
*   **Information Hiding (Principio Need to Know):** Linea guida di progettazione secondo cui le interfacce pubbliche espongono unicamente ciò che è strettamente necessario al chiamante, proteggendo gli attributi e nascondendo i dettagli implementativi interni.
*   **Javadoc:** Strumento adottato per generare e mantenere la documentazione direttamente nel codice sorgente del backend, descrivendo per ogni metodo responsabilità, parametri (`@param`), valori di ritorno (`@return`) ed eccezioni sollevate (`@throws`).
*   **JWT (JSON Web Token):** Standard crittografico adottato dal Security Layer per la gestione sicura delle sessioni autorizzate. I token sono firmati tramite algoritmo HMAC-SHA384 e la loro gestione rende il sistema completamente stateless.
*   **POJO (Plain Old Java Object):** Oggetti Java semplici che non dipendono da interfacce o framework esterni. Sono utilizzati per implementare tutti i DTO del sistema, spesso supportati dalla libreria Lombok per ridurre il codice boilerplate.
*   **Principal:** Oggetto popolato dal filtro di sicurezza JWT (JwtFilter) che incapsula l'identità verificata dell'utente. Viene iniettato nei Controller per autorizzare in modo sicuro le operazioni senza esporre dati sensibili nella richiesta.
*   **Repository:** Livello architetturale e pattern che incapsula la logica necessaria per accedere ai dati persistenti. In UniMove è implementato tramite interfacce *Spring Data JPA*, delegando al framework la generazione automatica delle query CRUD.
*   **Service (Service Layer):** Oggetto di soluzione che contiene, coordina e incapsula tutta la logica di business complessa dell'applicazione (es. `AuthService`, `RideService`). Viene invocato dai Controller e interagisce con i Repository o con sistemi esterni come CINECA.
*   **Stateless:** Modalità operativa del backend in cui il server non mantiene in memoria lo stato della sessione. Operazioni come il logout non modificano lo stato del server, ma avvengono tramite l'eliminazione del token lato client.
